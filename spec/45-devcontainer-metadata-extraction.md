# Devcontainer Metadata Extraction

## Purpose

Extract the `devcontainer.metadata` LABEL from `images/agents/Dockerfile` into
the template system, making the Dockerfile a pure container builder and the
shared devcontainer config a first-class template artifact.

## Problem

The `LABEL devcontainer.metadata` in `images/agents/Dockerfile` (lines 167-267)
embeds user-preference configuration — bind mounts for dotfiles, editor configs,
AI tool configs, terminal environment variables, and lifecycle hooks — directly
into a Docker image build file. This conflates two concerns:

1. **Image building**: installing packages, creating users, setting up the OS
2. **Devcontainer configuration**: mounts, env vars, remote user, lifecycle hooks

The Dockerfile should remain a pure container creator. Devcontainer
configuration belongs in the template system.

## Current Architecture

```text
┌─────────────────────────────────┐
│  Docker Image (agents)          │
│  LABEL devcontainer.metadata    │  ← shared config baked into image
│  (remoteUser, mounts, env, ...) │
└──────────────┬──────────────────┘
               │ Dev Container CLI merges at container-create time
               ▼
┌─────────────────────────────────┐
│  Template devcontainer.json     │  ← project-specific deltas only
│  (name, image, lang mounts)     │
└─────────────────────────────────┘
```

Templates are minimal because the LABEL provides shared defaults. The Dev
Container CLI merges image metadata with `devcontainer.json` automatically:
scalars use last-wins, objects merge by key, arrays append, lifecycle hooks
accumulate.

## Target Architecture

```text
Templates (scaffolding only)
────────────────────────────

  ~/.local/share/dctl/templates/_base/
  ~/.local/share/dctl/templates/<name>/
                    │
                    │ dctl init copies
                    ▼
Config (user SoT)
─────────────────

  ~/.config/dctl/devcontainer/_base/
  ~/.config/dctl/devcontainer/<name>/
                    │
                    │ dctl init merges
                    ▼
Cache (generated output)
────────────────────────

  ~/.cache/dctl/devcontainer/<name>/
  devcontainer.json
```

### Three-tier XDG layout

| XDG path | dctl usage | Mutable by |
|----------|-----------|------------|
| `~/.local/share/dctl/templates/` | Installed templates (`make install`) | `make install` only |
| `~/.config/dctl/devcontainer/` | Deployed config (seeded from templates, user-editable) | `dctl init` + user |
| `~/.cache/dctl/devcontainer/` | Generated merged configs | `dctl` (auto-generated) |

- **Data** (`~/.local/share`): installed source templates — lean, modular,
  scaffolding. `make install` writes here. Used only by `dctl init` to seed
  config.
- **Config** (`~/.config`): deployed config files — seeded from templates by
  `dctl init`, then user-editable. Contains separate `_base` and per-template
  files. This is the user's SoT.
- **Cache** (`~/.cache`): generated output — complete, self-contained
  `devcontainer.json` files. 100% derivable from config. Can be deleted and
  regenerated by `dctl init`. This is what `dctl ws up` actually consumes.

## Nomenclature Changes

The rename `base` → `general` improves clarity:

| Before          | After           | Role                                       |
|-----------------|-----------------|---------------------------------------------|
| `templates/base/` | `templates/general/` | User-selectable general-purpose template |
| *(new)*         | `templates/_base/`   | Internal shared config, never user-facing |

- `_base`: underscore prefix marks it as internal infrastructure. It is never
  listed by `dctl init --list`, never selectable interactively, and never
  deployed directly to a project. It serves only as the shared foundation that
  all templates compose with.
- `general`: the renamed user-facing template for generic projects using
  `devimg/agents:latest`. Formerly `base`.

All references in specs, Makefile, `template_registry_defaults()`, and docs
must be updated accordingly.

## Design Decisions

### Why not Dev Container `extends`?

The Dev Container specification does not support an `extends` or `inherits`
field. The only built-in composition mechanisms are image metadata LABELs
(what we are removing), Features (for tool installation), and Docker Compose.
Composition must therefore happen at the `dctl` tool level.

### Why `~/.cache` for generated output?

The merged `devcontainer.json` files are 100% derivable from the config layer
that `dctl init` seeds from installed templates. Per the XDG Base Directory
Specification, `XDG_CACHE_HOME` is for "non-essential data files" that "can be
deleted without loss of information" — the program can regenerate them.

This keeps config lean (only user intent) and separates machine-generated
artifacts from human-authored sources. Deleting `~/.cache/dctl/` is always
safe; `dctl init` regenerates the cache.

### Why deploy-time merge (not runtime)?

- `dctl ws up` passes `--config <path>` to the devcontainer CLI; that CLI does
  not know about `dctl`'s internal template structure
- Once the LABEL is removed, there is no other automatic injection mechanism
- Deploy-time merge produces inspectable, debuggable configs
- Users can edit their config files without understanding the merge internals
- `dctl ws up` does not regenerate cache; regeneration happens only in `dctl init`

### Why `_base` with underscore prefix?

The underscore prefix convention signals "internal, not user-facing". It
enables simple filtering in `discover_templates()` (`[[ "$name" == _* ]]`) and
prevents confusion with the user-selectable `general` template.

#### Template seeding and config SoT

Templates resolve from the installed templates directory only:

1. Installed `_base`: `~/.local/share/dctl/templates/_base/devcontainer.json`
2. Installed template: `~/.local/share/dctl/templates/<name>/devcontainer.json`

`dctl init` copies those files into the config layer:

1. Config `_base`: `~/.config/dctl/devcontainer/_base/devcontainer.json`
2. Config template: `~/.config/dctl/devcontainer/<name>/devcontainer.json`

After seeding, user customization happens only in the config layer. Templates
remain scaffolding only and are never consulted for runtime merge.

#### Merge function

Add `merge_base_and_template()` to `init.sh`:

```bash
merge_base_and_template() {
  local base_path="$1"
  local template_path="$2"

  jq -s '
    .[0] as $base | .[1] as $tmpl |
    $base * $tmpl |
    .mounts = (($base.mounts // []) + ($tmpl.mounts // [])) |
    .postCreateCommand = (($base.postCreateCommand // {}) * ($tmpl.postCreateCommand // {})) |
    .containerEnv = (($base.containerEnv // {}) * ($tmpl.containerEnv // {}))
  ' "$base_path" "$template_path"
}
```

Merge semantics (matching current LABEL merge behavior):

- `jq *` does recursive object merge with last-wins for scalars
- `mounts`: explicit array concatenation (base mounts + template mounts)
- `postCreateCommand`: object merge by key (base `dotfiles` preserved alongside
  template entries like `python`, `pre-commit`)
- `containerEnv`: object merge by key (template keys override base keys)

#### Cache invalidation

The generated config in `~/.cache/dctl/devcontainer/<name>/devcontainer.json`
is valid as long as its config source files have not changed. Invalidation uses
mtime comparison:

```bash
cache_is_fresh() {
  local cached_path="$1"
  local base_path="$2"
  local template_path="$3"

  [[ -f "$cached_path" ]] || return 1
  [[ "$cached_path" -nt "$base_path" ]] || return 1
  [[ "$cached_path" -nt "$template_path" ]] || return 1
}
```

If the cache is stale (or missing), `dctl init` regenerates it before use.
Freshness checks use config mtimes, not template mtimes. `dctl ws up` does not
regenerate cache.

The `--force` flag bypasses the freshness check and always regenerates.

#### Modify `deploy_template_config()`

In `lib/dctl/init.sh`, seed config first, then merge config to cache:

```bash
deploy_template_config() {
  local template="$1"
  local force="${2:-false}"

  local installed_base installed_tmpl
  installed_base="$(installed_template_path _base)"
  installed_tmpl="$(installed_template_path "$template")"

  local config_base config_tmpl
  config_base="$(config_devcontainer_path _base)"
  config_tmpl="$(config_devcontainer_path "$template")"

  _seed_config_file "$installed_base" "$config_base" "$force"
  _seed_config_file "$installed_tmpl" "$config_tmpl" "$force"

  cached_path="$(deployed_devcontainer_path "$template")"

  if [[ "$force" != true ]] && cache_is_fresh "$cached_path" "$config_base" "$config_tmpl"; then
    log "Using cached config: $cached_path" >&2
    printf '%s\n' "$cached_path"
    return 0
  fi

  mkdir -p "$(dirname "$cached_path")"
  merge_base_and_template "$config_base" "$config_tmpl" > "$cached_path"
  log "Generated config for '$template' at $cached_path" >&2
  printf '%s\n' "$cached_path"
}
```

The resolution chain in `common.sh` (`resolve_devcontainer_config()`) needs no
structural changes — it already follows the registered path. The only change is
that the path now lives under `~/.cache/`.

#### Migration concerns

Users with existing deployed configs (created before this change) will have
old configs at `~/.config/dctl/devcontainer/` (the former deploy target) and
stale registry entries pointing there. After rebuilding the image without the
LABEL, those configs will be incomplete.

Migration path:

- `dctl init --force --template <name>` regenerates the cached config and
  updates the registry entry to point to the new cache path
- Document this in release notes
- Consider detecting stale registry entries pointing to the old
  `~/.config/dctl/devcontainer/` path and warning the user

## Acceptance Criteria

### Existing criteria preserved

All 14 scenarios from `spec/99-acceptance-criteria.md` must continue to pass.
Key scenarios to verify:

- **Scenario 6** (template discovery): `_base` must NOT appear in
  `dctl init --list` output
- **Scenario 7** (coordinator template): cached coordinator config must
  contain both shared mounts (from `_base`) and Projects mount (from template)
- **Cross-cutting**: `make install` never overwrites `~/.config/dctl` files

### New criteria

- **N1**: `dctl init --template python` generates a cached config at
  `~/.cache/dctl/devcontainer/python/devcontainer.json` containing all 17 base
  mounts plus the poetry cache volume
- **N2**: `dctl init --template general` generates a cached config containing
  all shared settings (remoteUser, mounts, containerEnv, postCreateCommand)
- **N3**: Modifying a config file and running `dctl init` regenerates the
  cached config automatically (mtime invalidation against config mtimes)
- **N4**: `dctl init --force` always regenerates regardless of mtime
- **N5**: User edit at `~/.config/dctl/devcontainer/_base/devcontainer.json`
  is reflected in the generated cache
- **N6**: User edit at `~/.config/dctl/devcontainer/python/devcontainer.json`
  is reflected in the generated cache
- **N7**: Deleting `~/.cache/dctl/` entirely is safe; `dctl init` regenerates
  it
- **N8**: Docker image built from `images/agents/Dockerfile` has no
  `devcontainer.metadata` LABEL
- **N9**: Container behavior (mounts, env, lifecycle) is identical before and
  after the migration
- **N10**: `_base` never appears in `dctl init --list` or interactive template
  selection
- **N11**: `make install` writes only to `~/.local/share/dctl/`, never to
  `~/.config/dctl/` or `~/.cache/dctl/`

## Projects with local `.devcontainer/devcontainer.json`

Projects using a local devcontainer.json (not deployed via `dctl init`) that
currently rely on the LABEL for shared settings will need attention after
Phase 3. Options:

1. Run `dctl init` to get a properly merged cached config
2. Manually add shared settings to their local config

This is the most significant migration concern and should be documented in
release notes.

## Sequencing

Phases 1-2 can be implemented together (the merge function and the `_base`
template are co-dependent). Phase 3 (LABEL removal) should be a separate
commit after Phase 2 is verified working. Phase 4 runs alongside or after
Phase 3.

Recommended implementation order:

1. Phase 1 + Phase 2 (single PR)
2. Phase 3 + Phase 4 (follow-up PR, after verification)
